# An Absolute Beginner's Guide to npm and package.json

Using npm effectively is a cornerstone of modern web development, whether you're working exclusively with Node.js, using npm as a package manager or build tool for the front-end, or even incorporating it into workflows in other languages and on other platforms.

Really understanding npm as a tool — understanding its core concepts — can be difficult for a beginner. Many hours can be spent just trying to figure out small details that seem minor or are taken for granted by others.

This guide breaks down npm in detail, for those who are entirely new to Node.js, npm, and the surrounding ecosystem.

---

## Understanding `package.json`

As a general rule, any project using Node.js will need a `package.json` file.

### What is a `package.json` file?

At its simplest, a `package.json` file is a manifest of your project — it includes the packages and applications your project depends on, information about its source control, and specific metadata like the project's name, description, and author.

Let's break down the core parts of a typical `package.json` file.

### Metadata: `name`, `version`, `description`, `license`, and `keywords`

Inside a `package.json`, you'll almost always find metadata specific to the project — no matter if it's a web application, Node.js module, or plain JavaScript library. This metadata helps identify the project and gives users and contributors a baseline understanding of it.

Here's an example of how these fields look in a `package.json` file:

```json
{
  "name": "metaverse",
  "version": "0.92.12",
  "description": "The Metaverse virtual reality. The final outcome of all virtual worlds, augmented reality, and the Internet.",
  "main": "index.js",
  "license": "MIT"
}
```

A `package.json` file is always structured in JSON format, which allows it to be easily read as metadata and parsed by machines.

> 💡 **Tip:** If formatting a `package.json` file manually feels daunting, there's a handy command that will automatically generate a base file for you — see the [`npm init`](#using-npm-init-to-initialize-a-project) section below.

### Managing Dependencies: `dependencies` and `devDependencies`

The other major role of `package.json` is that it lists a project's **dependencies** — the modules the project relies on to function.

Listing dependencies in `package.json` matters for two reasons:

1. **Reproducible installs** — Running an install command inside a project installs all the dependencies listed in `package.json`, meaning modules don't need to be (and almost never should be) bundled with the project itself.
2. **Separation of concerns** — It allows a split between dependencies needed for _production_ and those needed for _development_. In production, you likely don't need a tool that watches your CSS files for changes. But in both environments, you'll want the modules that power your project — your web framework, API tools, and utilities.

Here's the previous example expanded to include dependencies:

```json
{
  "name": "metaverse",
  "version": "0.92.12",
  "description": "The Metaverse virtual reality. The final outcome of all virtual worlds, augmented reality, and the Internet.",
  "main": "index.js",
  "license": "MIT",
  "devDependencies": {
    "mocha": "~3.1",
    "native-hello-world": "^1.0.0",
    "should": "~3.3",
    "sinon": "~1.9"
  },
  "dependencies": {
    "fill-keys": "^1.0.2",
    "module-not-found-error": "^1.0.0",
    "resolve": "~1.1.7"
  }
}
```

One key difference between `dependencies`/`devDependencies` and the other fields is that both are **objects** with multiple key/value pairs. Every key is the name of a package, and every value is the version range that's acceptable to install, according to [Semantic Versioning (semver)](#).

## The Essential npm Commands

When using npm, you'll mostly interact with it through the command line. Below is a rundown of the commands you'll encounter and use most frequently.

### Using `npm init` to Initialize a Project

The `npm init` command is a step-by-step tool that scaffolds your project. It prompts for input in the following order:

1. The project's name
2. The project's initial version
3. The project's description
4. The project's entry point (its main file)
5. The project's test command (e.g., to trigger testing with something like Standard)
6. The project's git repository
7. The project's keywords (tags related to the project)
8. The project's license (defaults to ISC — most open-source Node.js projects use MIT)

If you're happy with the suggested default next to a prompt, just hit **Return**/**Enter** to accept it and move to the next one.

Once you complete the prompts, a `package.json` file is generated in the current directory. If you run it somewhere that isn't your dedicated project folder, don't worry — generating a `package.json` doesn't do anything beyond creating that file. You can move it afterward, or just create a new one in the right directory.

**Usage:**

```bash
npm init   # This will trigger the initialization
```

### Using `npm init --yes` to Instantly Initialize a Project

If you'd rather skip the prompts entirely, use the `--yes` flag to auto-populate all options with npm's default values.

> **Note:** You can configure what these defaults are via npm configuration — a more advanced topic beyond the scope of this guide.

**Usage:**

```bash
npm init --yes   # This will trigger automatically populated initialization
```

### Install Modules with `npm install`

Installing modules is one of the most fundamental things to learn with npm. Here's the core syntax for installing a standalone module into the current directory:

```bash
npm install <module>
```

Replace `<module>` with the name of the module you want to install. For example, to install [Express](https://expressjs.com/) (the most widely used Node.js web framework):

```bash
npm install express
```

This installs the `express` module into the `/node_modules` folder in the current directory. Whenever you install a module from npm, it lands in `node_modules`.

You can also install _all_ dependencies and devDependencies listed in a project's `package.json` by simply running:

```bash
npm install
```

**Alias:** `npm install` has a shorthand alias, `npm i`. This can trip up beginners, since there's no single standardized convention that maintainers use when documenting how to install their module.

**Usage:**

```bash
npm install <module>   # Where <module> is the name of the module you want to install
npm i <module>          # Same as above, using the "i" alias
```

### Install Modules and Save Them as a Dependency

Add the `--save` flag to `npm install` to add the module as an entry under `dependencies` in `package.json`.

**Usage:**

```bash
npm install <module> --save   # Where <module> is the name of the module you want to install
```

### Install Modules and Save Them as a Dev Dependency

The `--save-dev` flag works almost identically to `--save`, except the module is saved under `devDependencies` instead of `dependencies`.

The semantic difference: `dependencies` are needed in production, while `devDependencies` are only needed during development — things like testing tools, local dev servers, and similar utilities.

**Usage:**

```bash
npm install <module> --save-dev   # Where <module> is the name of the module you want to install
```

### Install Modules Globally

The final, and one of the most common, flags for `npm install` installs a module **globally** on your system rather than locally to the current directory. Use the `--global` flag (or its alias `-g`).

Global modules can be extremely useful — there are countless tools and utilities for both development and general use that make sense to install globally.

> **Note:** By default, npm installs global modules to a system directory, which typically requires elevated/privileged permissions. As a best practice, consider changing the default install location to a user directory instead.

**Usage:**

```bash
npm install <module> --global   # Where <module> is the name of the module you want to install globally
npm install <module> -g          # Same as above, using the "-g" alias
```
